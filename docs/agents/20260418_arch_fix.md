## 问题分析

本次修复涵盖了两个在 `ares-v4-engine` 核心链路中可能引发级联故障的架构级隐患：

1. **动态熵值溢出风险 (`src/engine/entropy.py`)**：
   - **需求本质**：在异常边缘测试（如遭遇大量伤停、极端战术惩罚叠加）中，计算出的动态熵值（`s`）以线性方式累加，存在直接突破 `1.0` 的风险。
   - **风险**：若不加设上限，该值可能会渗入下游计算环节（如生成 `EV` 概率）引发难以追踪的负数或超出定义域断言错误（System Crash / Model Hallucination）。
2. **本地数据隔离引发的数据滞后错觉 (`src/data/obsidian.py`)**：
   - **需求本质**：目前整个战术特征工程过度信赖本地 YAML 元数据。实战场景中，球队伤停状态可能在赛前最后几小时巨变。
   - **风险**：若本地无最新更新档案则会导致引擎基于“旧时态”得出错误推论（又称代理失忆效应/系统幻觉）。为将来整合真实数据流预装防线。

## 方案设计

### 1. 强制数值封顶机制 (Clamping)
- 通过在熵值各项惩罚最终核算结束后，统一加入限制 `s = min(s, 1.0000)`，形成数值安全阀，确保向下游系统输出的连续性与算法健壮性（Robustness）。

### 2. OSINT 隔离校验层
- 在反序列化并解析物理文件结构（Frontmatter 与 Body Yaml）与后续领域模型组装（生成 `TacticalProfile` 对象）中间强行切一刀，插入 `verify_with_osint` 网关功能拦截点。
- 目前配置为 Stub 模式（仅触发 Logger），保障了对现有接口零入侵，并为将通过诸如 API 取得外部信息做了无缝占位。

## 代码实现

### 1. 修复：S 值线性溢出 (`src/engine/entropy.py`)

在执行阈值熔断前，新增如下限制代码：
```python
    # 对动态熵值进行封顶，避免发生线性溢出
    s = min(s, 1.0000)

    # 5. 阈值熔断
    status = (
        "CRITICAL_WARNING"
        if s > inp.system_fragility_threshold
        else "STABLE"
    )
```
